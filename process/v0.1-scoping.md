# Swerdlow v0.1 — Scoping

A working doc, not a spec. Decisions captured from the 2026-05-11 design conversation. Pilot: **ContextDrift first**, designed to be Project B-compatible without painting us into a corner.

## Settled

### 1. Unindexed docs are not errors
- A `.md` file without frontmatter is a leaf the graph doesn't know about.
- No warnings, no validation failures.
- Bootstrap (later) can suggest frontmatter for un-indexed files. Authors opt in per file.
- Reason: Project B has ~194 `.md` files on the authoritative branch (`origin/working-docs-catchall-032626`). We will never have 100% coverage there. The tool must be useful at 5% coverage on day one.

### 2. Glob-based include / exclude — no `roots:` concept
The kickoff's `roots: [docs/]` bakes in a doc-directory assumption. Project B's docs are scattered: root-level `.md` (`AGENTS.md`, `CELERY_MIGRATION.md`, `121426_patch.md`, `GLOSSARY.md`, `MEMORY.md`), plus `docs/`, plus inline `core/monte_carlo/adapters/README.md`, etc. Config supports any layout:

```yaml
include:
  - "Docs/**/*.md"     # ContextDrift
  # or for Project B:
  - "**/*.md"
exclude:
  - "archive/**"
  - "node_modules/**"
  - ".git/**"
```

### 3. Explicit `depends_on:` per doc (no `core_docs:` yet)
- Hub docs (ContextDrift's `GDD`, `TDD`, `GamePillars`; Project B's `GLOSSARY`) appear in many `depends_on:` lists. Verbose but explicit.
- If repetition becomes a real pain, add `core_docs:` to config in v0.2+. Not before.
- Reason: fewer concepts is cheaper to teach and reason about.

### 4. `id` defaults to filename stem, override allowed
- `feature_spec.md` → `id: feature_spec` by default.
- Authors can set `id: parking` in frontmatter to override.
- Decouples graph identity from filename so renames don't shatter refs.

### 5. v0.1 CLI surface (minimal)
- `swerdlow bundle <id>` → ordered list of file paths (transitive closure of `depends_on`).
- `swerdlow graph` → render DAG (Graphviz + Mermaid).
- Nothing else. No MCP, no `reverse`, no `check`, no `bootstrap`.

### 6. Bundle output is just paths
- No formatting opinion in v0.1. Just newline-delimited file paths, ordered.
- Lets downstream tooling (CGDW skill, MCP server, whatever) shape it later without us guessing now.

## Open questions for next conversation

1. **Frontmatter required fields.** Kickoff proposes `id`, `status`, `type`. With #1 above (un-indexed is fine), is `status` actually required on indexed docs, or can it default? Same question for `type`. Minimum viable could be: `id` only, everything else optional.
2. **Transitive vs. direct deps in `bundle`.** Kickoff favors full transitive closure with `--depth` flag. Confirm for v0.1.
3. **Cycle behavior.** If A → B → A, what does `bundle A` return? Error, or break the cycle and continue with a warning? (Project B is likely to have accidental cycles.)
4. **Wikilinks / markdown links.** v0.1 reads only frontmatter `depends_on:`. We ignore `[label](path.md)` and `[[wikilinks]]` for now. Bootstrap (v0.3) will parse these. Confirm.
5. **`touches:` field.** Punt or include in v0.1 schema (read but unused)? Cheaper to define the field now and not use it than to migrate frontmatter later.

## Project B-compatibility guardrails (things v0.1 must NOT do)

- Don't assume a doc directory exists.
- Don't require every `.md` to have frontmatter.
- Don't couple `id` to filename irreversibly.
- Don't bake `bundle` output format into anything else (no premature MCP shape, no premature CGDW skill shape).

## Out of scope for v0.1 (explicitly deferred)

- MCP server (v0.2)
- `swerdlow reverse <path>` (v0.2)
- `swerdlow check` cycle/orphan/dead-ref detection (v0.2)
- `swerdlow bootstrap` auto-detect refs (v0.3)
- CGDW integration shape — skill vs. recipe vs. hook (decide after v0.2 lands)
- Status-aware bundling, watch mode, git hooks, Obsidian compat (v0.4+, speculative)

## Pilot validation plan

1. Build v0.1 against a fresh empty project — sanity-check the CLI surface.
2. Hand-add frontmatter to ~5 ContextDrift docs (likely candidates: `GDD.md`, `TDD.md`, `Now.md`, `Roadmap.md`, one deliverable spec).
3. Run `swerdlow bundle <one-deliverable-spec>` and check the output feels right.
4. If it does, expand to 15–20 docs and revisit before v0.2 design.
5. Only after ContextDrift feels solid do we touch Project B.
