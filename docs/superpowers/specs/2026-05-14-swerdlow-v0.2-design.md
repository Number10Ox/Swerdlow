v0.1 + v0.1.1 are shipped. Phase B (wire one consumer path in ContextDrift's SessionStart hook to call `swerdlow context`) is blocked until v0.2 ships a bundle-scoping primitive. v0.2 unblocks Phase B.

## 1. Purpose

v0.1's `depends_on:` is a flat untyped edge: every dependency is "always relevant." In a heavily-cross-linked corpus, transitive closure returns ~75% of the indexed set on most non-leaf targets, which is the bundle-ballooning problem ContextDrift's pilot surfaced.

v0.2 introduces **typed edges with free-form mode tags**. Each `depends_on:` entry can declare *when* the edge should be loaded:

```yaml
depends_on:
  - GamePillars                              # always relevant (v0.1 form preserved)
  - {id: MissionGameplay, when: [narration]} # narration sessions only
  - {id: PlanTemplate, when: [plan]}         # plan sessions only
```

Queries become mode-aware:

```bash
$ swerdlow context NarrativeDesign --for narration
```

This filters edges at every traversal hop, cutting branches that don't match the requested mode. The pilot's 23-file `context NarrativeDesign` bundle becomes ~7 files when the corpus is properly annotated.

## 2. User stories

The v0.1 user stories carry forward unchanged (retrofit, maintain over time, get the context bundle, surface in workflow artifacts). v0.2 adds two:

5. **Author scopes edges by work type.** When declaring `depends_on:` on a doc, I can annotate which kinds of sessions need each dependency. The annotation matches the prose convention I already use ("Load when..." columns in design doc tables).

6. **Consumer asks for the bundle relevant to a specific work type.** From a `/narration` session start, I ask `swerdlow context X --for narration` and get the narration-relevant slice of X's deps. From `/plan`, I get the plan-relevant slice. Same doc, different right answer.

**Explicitly NOT in v0.2's stories:**
- Indexing a different graph per mode (mode is a filter, not a separate graph).
- Inferring modes from prose during bootstrap (modes are user-authored).
- Conflict resolution when modes overlap (modes are independent free-form tags, no taxonomy).

## 3. CLI surface

```bash
# v0.1-compatible: no filter, all edges (unchanged)
$ swerdlow context NarrativeDesign

# v0.2 new: filter to single mode (plus always-edges)
$ swerdlow context NarrativeDesign --for narration

# multi-mode: union (narration OR plan OR always)
$ swerdlow context NarrativeDesign --for narration,plan

# v0.2 new: discover which modes exist in the corpus
$ swerdlow modes
narration       23 edges, 12 docs
plan             7 edges,  4 docs
design          12 edges,  8 docs
narrator-prompt  3 edges,  2 docs
```

The `--for` flag accepts a comma-separated list of mode tags. `swerdlow modes` tabulates the mode set actually present in the indexed corpus, sorted by edge count descending. `init`, `bootstrap`, `bootstrap --apply` are unchanged from v0.1.

## 4. Frontmatter schema

> **Canonical semantics:** An edge's `when:` is a session-type filter. The edge is loaded iff the caller's `--for` argument includes at least one of the listed modes, OR the `when:` key is absent (or the entry is a bare string). All other forms are errors.

`depends_on:` accepts a **heterogeneous list** where each entry is either a string or a dict. Both forms can coexist in the same list.

**String form (v0.1, preserved):**

```yaml
depends_on:
  - GamePillars
```

Interpreted as `{id: "GamePillars", when: []}` (always relevant).

**Dict form (v0.2 new):**

```yaml
depends_on:
  - {id: MissionGameplay, when: [narration]}
  - {id: SceneAuthoring, when: [narration, plan]}
  - {id: TDD, when: plan}        # convenience: single string coerced to [plan]
  - {id: GamePillars}            # no `when` = always (same as bare string)
```

**Schema rules:**

- `id` is required, must be a string.
- `when` is optional. May be a **non-empty** list of strings, or a single non-empty string (coerced to a one-element list), or **absent entirely**.
- "Always relevant" is expressed by **omitting `when:` entirely**, OR by using the bare-string form. These are the only two valid ways to declare always-load.
- `when: []` (empty list) is a **parse_error**, not an alternate spelling for "always." Reading `when: []` as "no modes match, never load" is the natural prose reading; reading it as "always" is the opposite. The spec forbids the ambiguity rather than picking one.
- Anything else (entry is a number / bool / dict without `id` / dict with malformed `when` / `when: []`) records an `Issue(type="parse_error")` and the entry is dropped.

**Mode vocabulary:** free-form. Swerdlow ships no known-modes enum. Each project defines its own (e.g., `narration`, `plan`, `design`, `test`). No validation against a known-modes list.

**Case sensitivity:** modes are case-sensitive strings (`narration` ≠ `Narration`). Documented; not auto-normalized.

## 5. Data model

New `Edge` dataclass replaces `tuple[str, str]`:

```python
@dataclass(frozen=True)
class Edge:
    from_id: str
    to_id: str
    when: tuple[str, ...] = ()   # empty tuple = "always"; otherwise mode tags
```

`tuple` (not `list`) so the dataclass stays frozen and hashable.

`Graph` updated:

```python
@dataclass(frozen=True)
class Graph:
    nodes: dict[str, Node]
    edges: list[Edge]              # was: list[tuple[str, str]]
    issues: list[Issue]
```

`Node`, `Issue`, `Proposal`, `BootstrapIssue`, `BootstrapPlan` are unchanged.

**`Issue.type` is unchanged.** Existing kinds (`missing_ref`, `duplicate_id`, `parse_error`, `cycle_detected`) carry over. Malformed dict entries fold into `parse_error`. No new issue kinds in v0.2.

## 6. Loader changes

`_build_edges` is rewritten to normalize each `depends_on` entry:

- String entry → `Edge(from_id, to_id=string, when=())`.
- Dict entry with valid `id` and optional `when` → `Edge(from_id, to_id=dict['id'], when=tuple(when_list))`.
- Malformed entry → `Issue(type="parse_error", doc_id=<from>, detail=<reason>)`.

Pass 1 (node build) is unchanged. Determinism guarantees (lexical path sort, first-wins on duplicate ID) carry forward.

## 7. Context query

`context(graph, doc_id, modes=None)` gains an optional `modes` parameter.

**Filter rule:**

An edge `Edge(from_id, to_id, when=W)` is **allowed** during traversal if:
- `modes is None` (no filter; v0.1 behavior), OR
- `W` is empty (always-relevant), OR
- `W` intersects `modes` (any tag matches).

Otherwise the edge is **cut** for this traversal, as if it didn't exist.

**Algorithm:**

1. Find target node by id. Not found → `ContextError` (CLI exits 1).
2. DFS from target. At each node, recurse into edges that pass the filter rule above (and only those).
3. Visit-set + ancestor-set tracking unchanged from v0.1. Cycles still produce `Issue(type="cycle_detected")` but only for edges that survived the mode filter. a cycle reachable only through filtered-out edges is invisible (desirable).
4. Post-order emit produces the topologically-sorted path list.

**Same-depth ordering** is preserved by walking the `depends_on:` list in declaration order, filtering as we go.

**Worked example. filter at every hop:**

```
NarrativeDesign --for narration:

  NarrativeDesign → [MissionGameplay {when:[narration]}, TDD {when:[plan]}]
                     ↓ kept                                ↓ cut
                     MissionGameplay → [InteriorAssets {when:[design]}]
                                                            ↓ cut
                     InteriorAssets does NOT enter the bundle, even though
                     nothing forbade it directly. its only entry path was
                     via an edge cut at hop 1 (NarrativeDesign→MissionGameplay
                     was narration-kept, but MissionGameplay→InteriorAssets
                     was design-only).

Result: NarrativeDesign + MissionGameplay (2 files).
```

This is the F1-killing behavior. Root-only filtering would have kept InteriorAssets (since the root edge survived); hop-by-hop filtering cuts the branch.

**Cycle reporting is mode-dependent.** `swerdlow context X` and `swerdlow context X --for narration` produce different cycle issue counts on the same corpus. a cycle that only exists via filtered-out edges does not appear under the filter. Users running both queries and grepping `cycle_detected` will see this. v0.3's `swerdlow check` will walk the unfiltered graph for canonical cycle reporting.

**CLI flag:**

```python
p_ctx.add_argument("--for", dest="for_modes", default=None,
                   help="Comma-separated mode tags to filter edges by")

# Parsed:
modes = args.for_modes.split(",") if args.for_modes else None
```

Modes are passed to `context(graph, doc_id, modes=modes)`. `None` preserves v0.1 behavior.

## 8. Apply + scan + plan file

**Apply:** the ruamel round-trip already handles arbitrary YAML structures byte-preservingly, so user-authored typed entries survive unchanged through apply cycles. New behavior: `add_depends_on` from the plan file can contain dicts (if the user hand-edited the plan to add `when:`). Each entry is written as-is. string stays string, dict stays dict, **flow-style stays flow, block-style stays block**. Idempotency continues to subtract by ID, ignoring mode tags. Both styles below are valid and must round-trip preserved:

```yaml
# Flow style
depends_on:
  - {id: MissionGameplay, when: [narration]}

# Block style
depends_on:
  - id: MissionGameplay
    when:
      - narration
```

**Scan:** unchanged user-facing behavior. Bootstrap still proposes bare-string entries from prose markdown links. Users add mode annotations during the plan-edit step (or after apply via direct edit of the frontmatter).

**Plan file:** `Proposal.add_depends_on: list[str | dict[str, Any]]`. heterogeneous list. PyYAML round-trips both forms transparently. No new plan-file schema.

## 9. CLI edge cases

**Unknown mode:** `--for nonexistent` does not error. Returns the always-only subgraph (which includes the target and any always-edges transitively). Stderr emits a one-time warning per unknown mode, listing the modes actually present in the corpus so the user can spot typos (case-sensitivity is a known foot-gun):

```
warning: mode 'narration' is not present on any edge.
         Modes used in this corpus: Narration, plan, narrator-prompt, design
```

The warning's mode list is computed in the same one-time pass that gathers `Edge.when` tuples for membership-check.

**Bootstrap doesn't update existing `when:` lists.** If a doc already declares `{id: Foo, when: [narration]}` and the user wants `when: [narration, plan]`, bootstrap will see `Foo` is already a declared dep and skip. proposing nothing. Modifying an existing edge's mode list requires direct editing of the frontmatter. Bootstrap is for *adding* edges, not for revising existing ones.

**Empty bundle from filter:** if a filter leaves the target with no edges and the target has no body content, just the target's path prints. Not an error.

**Mixed bare/typed entries in same frontmatter:** fully supported. The loader handles each entry independently.

## 11. Testing approach

`pytest`, fixture-driven, additive on top of v0.1's 65 tests.

**Loader tests:**
- Bare-string entries produce `Edge(when=())`. backward compatibility.
- Dict entry `{id: foo, when: [narration]}` produces `Edge(when=("narration",))`.
- Single-string `when: narration` coerces to `("narration",)`.
- **`when: []` → `parse_error` issue, entry dropped (NOT silently treated as always).**
- Missing `id` in dict → `parse_error` issue, entry dropped.
- Non-list, non-string `when` (e.g., `when: 42`) → `parse_error`.
- Mixed bare/typed in same `depends_on` works.

**Context tests:**
- No filter (`modes=None`) returns same bundle as v0.1.
- Single-mode filter returns target + always-edges + matching-mode edges.
- Multi-mode filter returns union of all matching modes.
- Edge cut at hop N stops traversal beyond it (the "filtering at every hop" rule from §7 worked example).
- Unknown mode returns always-only bundle AND prints warning listing known corpus modes.
- Mode filter interacts correctly with cycles (cycle only via filtered-out edge → no cycle issue).
- Same-depth declaration order preserved through filter.

**`swerdlow modes` tests:**
- Empty corpus → empty output, exit zero.
- Corpus with mixed typed/untyped edges → lists each mode with edge count and doc count, sorted by edge count descending. Untyped (always) edges are excluded from the listing.

**Scan tests:**
- Idempotency works against typed entries in existing frontmatter (re-scan doesn't re-propose).
- Re-scan on a doc whose existing `{id: Foo, when: [narration]}` is logically incomplete (user wants `[narration, plan]`) still does NOT propose. bootstrap only adds, doesn't revise. Documented in §9.

**Apply tests:**
- Typed entries in source frontmatter survive a round-trip apply (byte-preserving), **both flow-style and block-style**.
- Plan with dict entry in `add_depends_on` writes a dict into source frontmatter.
- Plan with mixed string/dict entries writes both forms correctly.

**E2E test:**
- Extension of the v0.1 e2e test: same workflow, but the apply step writes some typed entries and `context X --for <mode>` returns the expected scoped bundle.

No changes to install, packaging, dependencies, or Python version requirement. v0.2 = v0.1.1 + the changes above.

## 13. Settled decisions (from brainstorm)

- **Primary mechanism: typed edges (per-edge metadata), not doc-level scope tags.** Mirrors the user's existing prose-table convention (F6).
- **Schema: `{id, when}` only. no `role` / `note` fields.** YAGNI; no consumer needs them yet.
- **`when` semantics: OR over modes.** `when: [narration, plan]` matches `--for narration` and `--for plan`.
- **CLI flag: `--for narration,plan` comma-separated.** One flag, multi-value.
- **Modes are free-form strings, project-defined.** No enum, no validation.
- **Filtering applies at every traversal hop, not just the root.** F1 is solved by aggressive cutting.
- **v0.1 schema fully backward-compatible.** Bare-string entries are valid, always-relevant. No forced migration.
- **Data model: `Edge` dataclass.** Replaces tuples. Type-safe and frozen.
- **No new `Issue` kinds.** Malformed typed entries (including `when: []`) fold into `parse_error`.
- **`when: []` is a parse_error, not an alternate "always".** Only two valid spellings for always: bare string, or `{id: foo}` with no `when` key. Closes a semantic-ambiguity foot-gun.
- **Mode discovery via `swerdlow modes`** ships with v0.2 (originally considered v0.3). Without it, mode tags are write-only.
- **Unknown-mode warning lists corpus modes**, not just "unknown mode X." Helps catch typos and case-sensitivity errors.
- **Scope: typed edges + mode scoping + `swerdlow modes` discovery.** Visualize / reverse / check / MCP defer to v0.3.

## 14. Out of scope for v0.2

Deferred to v0.3+ (no order implied):

- Graph rendering (`swerdlow graph` → Graphviz / Mermaid output).
- Reverse lookup (`swerdlow reverse <code-path>` → which docs govern this file).
- `swerdlow check` (cycle detection, orphan docs, dead-ref report. useful but distinct from query-time issues).
- MCP server.
- LLM-priming surface (skill / plugin that primes agents on Swerdlow usage in a project).
- Indirection mechanisms for F2 (Now.md deps rot weekly). Mode scoping *may be enough*. `Now.md` could `depends_on: [{id: MissionGameplay, when: [del-046]}, {id: ResearchSynthesis, when: [del-045]}]` with the active-deliverable ID passed as `--for`. Phase B is the test: if that pattern works for the SessionStart hook, F2 doesn't need a separate primitive in v0.3. If it doesn't (e.g., the deliverable-ID-as-mode-tag approach scales poorly with deliverable churn), F2 gets its own design cycle.
- Index-vs-dep heuristic (B3 from pilot). Possible add via "links in tables / under specific headings = candidate suppressions."
- Bootstrap heuristics beyond markdown links (wiki-syntax, prose patterns, bare filenames).
- Per-section deps. Big architectural change; defer.

## 15. Open questions

None at design time. The pilot surfaced enough signal that all major decisions could be settled in the brainstorm + ContextDrift review pass. Surface any during implementation as code comments tagged `# OPEN-QUESTION-v0.2:` for v0.3 review.

**Forward-compatible additions considered for v0.3** (mentioned here so the v0.3 brainstorm has the input):
- **`note:` field on edges**. captures the prose-table "What it defines" column (F6's column 2). Lets a future consumer surface "why this file is in the bundle." Schema is forward-compatible. Swerdlow ignores unrecognized fields, so adding `note:` later doesn't break v0.2 frontmatter. Defer until Phase B reveals whether rationale strings are wanted in real consumer surfaces.

## 16. Validation plan

Before declaring v0.2 done:

1. **Self-test in `/Users/jedwards/workspace/Swerdlow`**. all v0.1 tests pass + new v0.2 tests pass. Backward compatibility verified.

2. **Re-pilot annotation on ContextDrift.** Annotate the already-frontmatter'd 25 docs with `when:` tags based on the existing "Load when..." prose tables. Workflow recommendation:
   - **For docs with a "Related docs (load when touching that area)" prose table:** translate the table directly. Column 3 ("Load when…") maps to `when:` tags; pick concise tag names (e.g., "Checking how narration fits in the beat cycle" → `when: [narration]`).
   - **For docs without such a table:** default to bare-string (always-relevant) until real usage tells you which edges are mode-specific. Annotation is iterative, not all-or-nothing.
   - Cost scales linearly with edge count: ~25 docs × ~7 deps avg ≈ ~175 entries to consider. Most will stay bare-string; the typed conversions happen where the prose tables already provide the answer.

3. **Phase B execution. measurable success criteria.** Wire `swerdlow context <doc> --for <session-mode>` into ContextDrift's SessionStart hook or a session-entry skill. v0.2 ships only if **all** of the following hold:
   - `swerdlow context NarrativeDesign --for narration` returns ≤ 10 files (target ~7).
   - That bundle **contains:** NarrativeDesign, MissionGameplay, MissionIntelModel, SceneAuthoring, AgentVoiceContract-Psalm.
   - That bundle **does NOT contain:** InteriorAssets, CCTVSignalGame, CityMap, MissionGeneration.
   - `swerdlow context Now --for narration` returns < 12 files (half of v0.1's 24).
   - Query latency on the ContextDrift corpus stays under 500 ms.

4. **If Phase B success criteria met:** v0.2 ships. Capture observations, decide v0.3 priorities from real usage. In particular:
   - Was the deliverable-ID-as-mode-tag pattern usable for F2? If yes, F2 doesn't need a separate v0.3 primitive.
   - Did consumers want a `note:` field? If yes, add in v0.3 (forward-compatible).
   - Did mode-discovery via `swerdlow modes` get used? If not, the listing format may need rework.

5. **If Phase B fails any criterion:** v0.2 is not done. Investigate which criterion failed, design the fix (may require schema or query-model adjustment), brainstorm v0.3 from the new pilot evidence.

## 17. Mode-tag conventions (recommendation, not enforcement)

Swerdlow's schema is free-form, but a project that picks conventions early will get less churn. Recommended conventions:

- **Use lowercase, kebab-case for mode tags.** Examples: `narration`, `plan`, `design`, `narrator-prompt`. (Swerdlow doesn't enforce, but the unknown-mode warning will surface typos faster if everyone's consistent.)
- **Tag by work-type, not by deliverable ID**. unless deliverable IDs are intentionally being used as time-bounded modes (the F2 experiment in §14). Work-type tags age well; deliverable-ID tags rot.
- **Common base set to start with:** `narration`, `plan`, `design`, `test`, `review`. Add project-specific ones as patterns emerge.
- **`--for <nothing>` (omit the flag) is the right escape hatch.** When in doubt about which mode applies, run without `--for` and get everything. Mode tags are an optimization, not a gate.
- **Mode tag stability is a soft contract.** Renaming a mode means hand-editing every doc that uses it. Pick names you'll keep.

These are recommendations, not part of the loader/query semantics. They live in this section so the project has a starting point if it adopts Swerdlow without prior conventions.
