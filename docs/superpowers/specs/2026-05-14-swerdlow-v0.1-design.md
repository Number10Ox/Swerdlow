# Swerdlow v0.1 — Design

**Date:** 2026-05-14 (revised after first review)
**Status:** Draft v2 — incorporates ChatGPT first-pass review (write-safety, deterministic loading, noisy-but-nonfatal issue reporting, drop networkx).
**Authors:** Jon (project owner), Claude (brainstorm partner).
**Context docs:** [`docs/kickoff.md`](../../kickoff.md), [`process/v0.1-scoping.md`](../../../process/v0.1-scoping.md).

This is the v0.1 design that comes out of the Superpowers `brainstorming` skill cycle. It supersedes the speculative roadmap in the kickoff at the v0.1 boundary; later versions remain as sketched there.

---

## 1. Purpose

Swerdlow is a markdown-to-markdown dependency layer for AI-assisted codebases. It indexes a corpus of markdown documents, builds a dependency graph from declared frontmatter, and answers: **"what files should I load to work on document X?"**

The intended consumers are LLM sessions (Claude Code, etc.) and the workflow layers around them (skills, planning tools, post-execution reports). Swerdlow itself produces the dependency-resolved file list; the workflow layer formats and embeds it.

See `docs/kickoff.md` for the broader project framing (what Swerdlow is NOT, prior art, pilot consumers, the v0.2+ roadmap).

## 2. User stories

User stories drive what's in v0.1; the schema is derived from them.

1. **Retrofit an existing corpus.** I have a project with many markdown docs that do not currently declare dependencies. I run a command that scans the corpus, proposes dependencies (where it can detect them via existing markdown links), and lets me review/edit the proposals before they're written into my files.

2. **Maintain dependencies over time.** When I add or change a doc, I update its `depends_on:` frontmatter either by hand or by re-running bootstrap (which is idempotent). Authoring by LLM-with-project-context is supported via the same plan-file mechanism — that LLM writes the plan, Swerdlow applies it. *(Note: the LLM-priming surface — skill/plugin/MCP — is a separate later cycle, not v0.1.)*

3. **Get the context for a doc.** Given a doc id, I get an ordered list of files (the doc itself plus its transitive `depends_on` deps, deepest first).

4. **Surface that context in workflow artifacts.** A skill, planning tool, conversation report, or post-execution summary takes Swerdlow's output and embeds it where the human or LLM will see it. Swerdlow produces the clean list; the workflow layer formats.

**Out of v0.1, explicitly:**
- Graph rendering (Graphviz / Mermaid) → v0.2.
- Reverse lookup ("which docs govern this code file?") → v0.2.
- `swerdlow check` for cycles / orphans / dead refs → v0.2.
- MCP server → v0.2.
- Bare-filename mentions, prose patterns ("see also"), code-file references in bootstrap → v0.3.
- Status-aware bundling, watch mode, git hooks, Obsidian compat → v0.4+.

## 3. CLI surface

```bash
$ swerdlow init
# Creates .swerdlow/config.yaml with default include/exclude globs.
# Refuses to overwrite an existing config; pass --force to regenerate.

$ swerdlow bootstrap
# NON-MUTATING to source files. Scans corpus, writes .swerdlow/bootstrap.plan.yaml.
# Refuses to overwrite an existing plan file; pass --force to regenerate.

$ # User edits the plan file by hand, or hands it to an LLM in the project
$ # for enrichment, then:

$ swerdlow bootstrap --apply
# MUTATING. Reads the plan file, writes depends_on: into each target file's frontmatter.

$ swerdlow context <doc-id>
# Prints ordered list of file paths to stdout, one per line.
# Includes the target doc itself plus all transitive depends_on deps,
# topologically sorted (deepest deps first, target last).
# Prints any graph issues (missing refs, duplicate ids, cycles detected) to stderr.
```

That is the entire v0.1 CLI. No graph rendering, no MCP, no reverse, no check.

**Help text discipline.** The two `bootstrap` invocations differ only in `--apply`. Plain `bootstrap` writes only to `.swerdlow/bootstrap.plan.yaml` (a fresh artifact, never your source files). `bootstrap --apply` writes to source files. The help text for both must make this distinction the first thing the user reads.

## 4. Configuration

`.swerdlow/config.yaml`:

```yaml
include:
  - "Docs/**/*.md"
exclude:
  - "node_modules/**"
  - "archive/**"
```

**Glob-based, no directory assumption.** The kickoff originally proposed a `roots:` concept; that was dropped because Project B-style scattered docs (root-level `.md`, plus `docs/`, plus inline `core/.../README.md`) don't fit a single-directory model. Glob include/exclude handles both ContextDrift (narrow `Docs/**/*.md`) and Project B (broad `**/*.md` plus aggressive exclude) without per-project forks.

`swerdlow init` writes a starter config; the user customizes from there.

## 5. Data model

### 5.1 In-memory graph types

```python
@dataclass(frozen=True)
class Node:
    id: str                          # filename stem unless overridden by explicit id:
    path: Path                       # path to the file
    frontmatter: dict[str, Any]      # full parsed YAML — depends_on plus any other fields

@dataclass(frozen=True)
class Issue:
    type: str                        # "missing_ref" | "duplicate_id" | "parse_error" | "cycle_detected"
    doc_id: str                      # the doc where the issue surfaced (or "" for cross-file issues)
    detail: str                      # human-readable; includes paths for path-level issues

@dataclass(frozen=True)
class Graph:
    nodes: dict[str, Node]           # id -> Node
    edges: list[tuple[str, str]]     # (from_id, to_id), declared depends_on edges
    issues: list[Issue]              # collected during load
```

All frozen. **No methods on `Graph`** — operations (`context`, future `check`, future `reverse`) are standalone functions that take a `Graph` as input. This keeps each operation independently testable and avoids a god-class as v0.2+ features land.

**No external graph library.** v0.1 hand-rolls the adjacency dict, DFS with visit-set, and post-order topological emit. Total ~30 lines. `networkx` was considered and rejected: it would pull scipy/numpy transitively (~50MB+) for a 50–200-node graph. Embarrassing overkill at this scale.

### 5.2 Bootstrap plan file types

```python
@dataclass(frozen=True)
class Proposal:
    file: Path
    add_depends_on: list[str]

@dataclass(frozen=True)
class BootstrapIssue:
    file: Path
    link: str
    detail: str

@dataclass(frozen=True)
class BootstrapPlan:
    proposals: list[Proposal]
    issues: list[BootstrapIssue]
```

Deliberately distinct from `Graph.issues`. Bootstrap issues are link-resolution problems written to a YAML file; graph issues are graph-loading problems held in memory. Different timing, different artifact, different problem class — sharing a type would conflate them.

### 5.3 Frontmatter schema (read by loader)

A markdown file is **indexed** if and only if it has a frontmatter block that parses cleanly as YAML. Three distinct outcomes:

- **No `---` block at file start:** silently unindexed. Not an error. The file is invisible to the graph.
- **`---` block present, parses cleanly (even if empty):** indexed.
- **`---` block present but malformed YAML:** unindexed (the file can't be loaded), but an `Issue(type="parse_error", doc_id=<filename>, detail=<parser error>)` is recorded. Distinct from the no-block case — surfaces actual damage rather than hiding it.

Recognized fields:

- `id` *(optional)* — explicit id. Defaults to filename stem (`MissionDesign.md` → `MissionDesign`).
- `depends_on` *(optional)* — list of ids the doc depends on for context.
- Any other fields *(passthrough)* — `status`, `type`, `title`, `touches`, `owners`, `updated`, `supersedes`, etc. Stored verbatim in `Node.frontmatter`, not validated or acted on in v0.1. Future versions may use them.

**Nothing is strictly required to type.** The frontmatter block itself is the only required marker.

### 5.4 ID model

- IDs default to filename stem.
- Filenames *should* be unique across the indexed corpus. The OS does not enforce this — the user must.
- Filename collisions (e.g., multiple `README.md` in different subdirs) are resolved by:
  1. Excluding the unwanted ones via config (`exclude: ["**/README.md"]`).
  2. Adding an explicit `id:` override to frontmatter on the conflicting file.
  3. Renaming a file.
- At load time, if two docs end up with the same id: first one wins (where "first" is defined by lexically-sorted path order — see §7); an `Issue(type="duplicate_id", detail="<winning-path> kept; <losing-path> shadowed")` is appended. Both paths in the detail so the user can resolve via exclude config, explicit `id:`, or rename without grepping.

## 6. Bootstrap workflow

### 6.1 Scan phase — `swerdlow bootstrap`

1. Read `.swerdlow/config.yaml` for include/exclude globs.
2. Walk the filesystem; collect matching `.md` files into the indexed set.
3. For each file:
   a. Read its contents.
   b. Parse markdown with a proper parser (`mistune`) to extract `[label](path.md)` links. Using a parser rather than regex avoids false positives inside code blocks, inline code, or comments.
   c. For each link:
      - URL (`http://...`, `https://...`) or non-markdown extension → silently skip.
      - Resolve to basename stem (`docs/foo.md` → `foo`; `../foo.md` → `foo`).
      - Target file not in the indexed set → record as `BootstrapIssue` (detail: `"target outside indexed corpus"` or `"target file does not exist"`).
   d. Parse existing frontmatter (if any); get current `depends_on:` set.
   e. Subtract existing deps from proposed (idempotency).
   f. If anything new remains, create a `Proposal`.
4. Write `.swerdlow/bootstrap.plan.yaml`:

   ```yaml
   proposals:
     - file: Docs/MissionDesign.md
       add_depends_on:
         - GamePillars   # from link: [pillars](GamePillars.md)
         - TDD           # from link: [tech doc](../TDD.md)

   issues:
     - file: Docs/SignalGame.md
       link: ../legacy-notes.md
       detail: target outside indexed corpus

     - file: Docs/GDD.md
       link: ./missing.md
       detail: target file does not exist
   ```

### 6.2 User edit step (out of band)

The user opens the plan file in their editor and:
- Removes proposals they don't want.
- Adds proposals the scan didn't catch (e.g., implicit dependencies that aren't expressed as standard markdown links — common in retrofit on existing corpora).
- Resolves issues by moving files into the indexed corpus, fixing the link, or leaving as intentional.

Plan file format is plain YAML — also writable by an LLM session that has the project loaded. This is the intended bridge for projects whose existing docs reference each other in prose ("see GDD") rather than as standard markdown links: run scan to get a skeleton, hand it to an LLM-in-project to enrich, apply the enriched plan.

### 6.3 Apply phase — `swerdlow bootstrap --apply`

1. Read `.swerdlow/bootstrap.plan.yaml`.
2. For each proposal:
   a. Read target file as text.
   b. Locate the frontmatter block by text (regex on `^---\n...\n---\n` at file start). Parse it with `ruamel.yaml` (round-trip mode) into a structure that preserves comments, key order, quoting style, and indentation.
   c. If no frontmatter block exists, prepare to prepend a new minimal block.
   d. Locate the existing `depends_on:` sequence (if any). Append new ids to it — existing entries first, then new entries, no reordering of existing entries.
   e. Re-serialize via `ruamel.yaml` round-trip and splice back into the file text. Preserve the original body verbatim (everything after the frontmatter block).
3. Report a summary count.

**Byte-preserving discipline is a v0.1 requirement, not an open question.** A retrofit tool that silently reformats user YAML (sorts keys, normalizes quotes, drops comments, changes indentation) loses trust the first time a user inspects a diff. `ruamel.yaml` round-trip mode is the standard solution. The cost is a slightly heavier library and more careful code; the benefit is that `git diff` after `--apply` shows only the lines that actually changed.

### 6.4 Bootstrap edge cases

- **No frontmatter block.** Prepend `---\ndepends_on:\n  - foo\n  - bar\n---\n` to the file.
- **Frontmatter exists, no `depends_on:` key.** Add the field; preserve all other fields.
- **Frontmatter exists with `depends_on:`.** Append new ids non-destructively; preserve order.
- **Apply is idempotent.** Re-applying the same plan against an already-applied corpus is a no-op (each `add_depends_on` entry is only added if not already present).
- **Plan references a file deleted/moved between scan and apply.** Log a warning, skip that proposal, continue with the others. Do not abort the whole apply.
- **Plan references a file outside the include globs.** Log a warning, skip. Plan can only act on files Swerdlow considers part of the corpus.

## 7. Load workflow

Used by every read command (today: `context`; later: `check`, `reverse`, graph rendering).

```python
def load_graph(config_path: Path | None = None) -> Graph: ...
```

1. Read `.swerdlow/config.yaml` for include/exclude globs.
2. Walk filesystem; collect matching `.md` files. **Sort the collected paths lexically before the build-nodes pass.** This makes "first-wins" semantics on duplicate ids reproducible across runs and platforms; without the sort, `os.walk` order is undefined and "first" is meaningless.
3. **First pass — build nodes:**
   - For each (sorted) file, read frontmatter with `python-frontmatter`.
   - No `---` block at file start → skip silently (unindexed, not an error).
   - `---` block present but malformed YAML → append `Issue(type="parse_error", doc_id=<filename>, detail=<parser msg>)`; file is unindexed. Distinct from the no-block case.
   - Block parses cleanly → determine id: explicit `id:` field if present, else filename stem.
   - Id collides with an already-added node → keep first (lexically earlier path), append `Issue(type="duplicate_id", detail="<winning-path> kept; <losing-path> shadowed")`. Both paths surfaced in the detail.
   - Otherwise: build `Node(id, path, frontmatter)`, add to graph.
4. **Second pass — build edges:**
   - For each node, iterate `frontmatter.get("depends_on", [])`.
   - For each entry, look up the target id in `nodes`.
   - Found → add edge `(from_id, to_id)`.
   - Not found → append `Issue(type="missing_ref", doc_id=<from_id>, detail="depends_on '<missing-id>' has no indexed target")`, no edge added.
5. Return `Graph(nodes, edges, issues)`.

**No cycle detection at load time** in v0.1. Cycles are real (Project B-scale corpora will accidentally produce them) and dedicated cycle reporting belongs in `swerdlow check` (v0.2). Consumers must be cycle-safe regardless — see `context` below.

## 8. Context query

```python
def context(graph: Graph, doc_id: str) -> list[Path]: ...
```

1. Find the target node by id. Not found → CLI exits 1 with an error message to stderr.
2. Walk `depends_on` edges depth-first from the target. Maintain a visit-set so cycles can't infinite-loop. **When the visit-set rejects a re-visit, a cycle edge has been crossed** — record an `Issue(type="cycle_detected", doc_id=<from-node>, detail="cycle edge into <re-visited-node>")` for each such occurrence. The traversal continues with that edge dropped. At each node, recurse into deps in the order they appear in the source `depends_on:` list (preserving authorial intent, keeping output deterministic).
3. Emit each node on post-order (after all its deps have been emitted). This produces a topological order: deepest deps first, the target doc last. Same-depth ties are broken by declaration order in the parent's `depends_on:`.
4. Return the ordered list of `Path` objects.

**CLI output:**

- File paths to stdout, one per line.
- All issues from the load pass (missing refs, duplicate ids, parse errors) **plus** any cycle-detected issues from this traversal, printed to stderr — one issue per line, prefixed by issue type. Loud-but-nonfatal: the path list still ships, the user sees what's wrong.
- **Exit zero** when the path list was produced successfully, regardless of issue count. Reserved nonzero exits: target id not found (1), config file missing or corrupt (2), plan file required-but-missing for apply (3). Loud nonzero exit codes are a v0.2 `check` concern; `context` is for the common-path use, where issues are expected.

No formatting opinion in v0.1; downstream consumers (workflow skills, planning tools, MCP server in v0.2) format the path list as needed.

**No `--depth N` flag in v0.1.** Default is unbounded transitive closure. A depth flag is a small later add if real use cases want it.

## 9. Module structure

```
swerdlow/
  __init__.py              # public exports: load_graph, context, types
  cli.py                   # argparse, command dispatch (init, bootstrap, context)
  config.py                # config file load + validation
  types.py                 # Graph, Node, Issue, Proposal, BootstrapPlan, BootstrapIssue
  loader.py                # load_graph()
  context.py               # context(graph, id) -> list[Path]
  bootstrap/
    __init__.py
    scan.py                # generate the plan file
    apply.py               # apply the plan file
    plan.py                # plan file (de)serialization
tests/
  fixtures/                # small hand-crafted .md trees
    simple-graph/
    cycle/
    missing-ref/
    duplicate-id/
    no-frontmatter/
    malformed-yaml/
    bootstrap-greenfield/
    bootstrap-idempotent/
  test_loader.py
  test_context.py
  test_bootstrap_scan.py
  test_bootstrap_apply.py
```

Boundaries:
- `loader.py` does not import `bootstrap/`. The read side and the write side are independent.
- `types.py` is the shared vocabulary; everything else depends on it.
- `cli.py` is the only place that does I/O via stdout/stderr; the library functions return values.

## 10. Testing approach

`pytest`, fixture-driven. Each fixture is a small hand-curated `.md` tree under `tests/fixtures/`. Each test loads a fixture, invokes one function, and asserts on the returned data structure.

Key cases:

- **Loader:**
  - Simple graph (happy path).
  - Missing ref produces `Issue(type="missing_ref")`; no edge added.
  - Duplicate id: first-wins by *lexically-sorted path order*, `Issue(type="duplicate_id")` records both paths. Test must verify reproducibility across two runs.
  - Files with no frontmatter block are skipped silently.
  - Files with a frontmatter block but malformed YAML produce `Issue(type="parse_error")`. Distinct from the no-block case.
  - Paths are deterministically ordered: feed the same fixture twice, get byte-identical `Graph.nodes` dict order.
- **Context:**
  - Direct deps, transitive deps, multi-level deps (deepest-first emit order).
  - Cycle is traversed safely (no infinite loop) AND emits `Issue(type="cycle_detected")` to the returned issue list. CLI must print it to stderr.
  - Missing target id → exit code 1, stderr message.
  - Same-depth ties broken by declaration order in `depends_on:`.
- **Bootstrap scan:**
  - Detects standard markdown links via `mistune` parser (not regex), no false positives from links inside code blocks or inline code.
  - Resolves paths to basename stem.
  - Marks out-of-scope and broken links as `BootstrapIssue` entries.
  - Idempotent: re-running against an already-applied corpus produces zero new proposals.
  - Refuses to overwrite an existing plan file without `--force`.
- **Bootstrap apply:**
  - Writes frontmatter to greenfield files (no prior block).
  - Appends to existing `depends_on:` non-destructively.
  - **Byte-preserving:** for a file with non-trivial frontmatter (key order, quoting style, comments, indentation), apply changes only the `depends_on:` lines — verified by diffing before/after.
  - Preserves other frontmatter fields verbatim.
  - Idempotent re-apply produces zero diff.
  - Gracefully handles a deleted target between scan and apply: logs and continues with remaining proposals.

## 11. Distribution

- `pyproject.toml` declares dependencies and the entry point `swerdlow = "swerdlow.cli:main"`.
- `uv tool install swerdlow` (or `pipx install swerdlow`) installs the CLI globally.
- Python 3.11+ for modern dataclass and typing features.
- Direct runtime dependencies: `python-frontmatter` (read), `ruamel.yaml` (byte-preserving round-trip for apply), `mistune` (markdown link extraction), `PyYAML` (plan-file (de)serialization only — plan file is generated by us, no preservation needs), `pathspec` (gitignore-style globs).
- No `networkx`. Hand-rolled adjacency dict + DFS + post-order topo sort, ~30 lines. See §5.1.
- Dev dependencies: `pytest`.

## 12. Settled decisions (carried in from scoping)

These were settled in `process/v0.1-scoping.md` or during the brainstorming cycle, and are listed here for reviewer convenience:

- **Glob-based `include`/`exclude` in config.** No `roots:` concept. No directory assumption.
- **Unindexed files are not errors.** Partial coverage is the design assumption; full coverage is never required.
- **`id` defaults to filename stem; explicit `id:` is escape hatch.** Filename collisions are user-resolved via exclude config, explicit override, or rename.
- **No fields are strictly required.** The frontmatter block itself (even empty `---\n---`) is the opt-in marker.
- **Approach A (dataclass + standalone functions), not class-with-methods.** Keeps each operation independently testable, avoids god-class growth.
- **Cycle behavior: don't detect at load; consumers are cycle-safe.** Dedicated cycle reporting belongs in v0.2 `check`.
- **Missing references → drop edge + record `Issue`.** Don't error; preserve signal for v0.2 `check`.
- **Duplicate ids → first-wins + record `Issue`.** Same pattern.
- **Bootstrap is bootstrap-first.** Without this, v0.1 cannot be tested on real corpora (which is why it was promoted from v0.3 to v0.1).
- **Bootstrap is batch-with-editable-plan-file**, not per-file interactive. User edits the YAML between scan and apply.
- **Existing frontmatter is preserved non-destructively** in apply.
- **`context` is the command name**, not `bundle` (closer match to user-story-2 framing, target doc is part of the context so no semantic mismatch).

## 13. Out of scope for v0.1

Reiterated for emphasis since several were considered and explicitly deferred:

- Visualization (Graphviz/Mermaid) — moves to v0.2.
- Reverse lookup, `check` (cycles/orphans/dead refs), MCP server — v0.2.
- Advanced bootstrap heuristics (bare filename mentions, "see also" prose patterns, wiki-syntax `[[brackets]]`, code-file references) — v0.3.
- Status-aware bundling, watch mode, git hooks, Obsidian compat — v0.4+.
- An LLM-priming surface (Swerdlow skill, plugin, or MCP server that primes an agent on how to use Swerdlow in a project) — separate later brainstorm cycle, not gated on v0.1.

## 14. Open questions for review

The first review cycle (ChatGPT) closed all five v1 open questions. Resolutions folded into the spec body. Recorded here for traceability:

1. **`networkx` as an internal dep.** RESOLVED → drop. Hand-rolled adjacency + DFS + post-order is ~30 lines and avoids pulling scipy/numpy transitively. See §5.1, §11.

2. **Frontmatter write format on apply.** RESOLVED → byte-preserving via `ruamel.yaml` round-trip is a v0.1 *requirement*, not an option. Silently reformatting user YAML on retrofit destroys trust. See §6.3, §10.

3. **Plan-file path hard-coded.** RESOLVED → keep hard-coded for v0.1. `--plan <path>` cut as YAGNI. Two walk-throughs (fresh start, update after add/delete) showed no concrete need. Trivial to add later if a real pilot scenario surfaces.

4. **`swerdlow init` / `swerdlow bootstrap` overwrite behavior.** RESOLVED → refuse to overwrite an existing config or plan file; require `--force` to regenerate. Friendlier failure mode than silent no-op or silent destruction. See §3.

5. **CLI exit codes.** RESOLVED → `context` exits zero when the path list ships, regardless of issue count. Issues print to stderr. Reserved nonzero codes are for genuine errors (target not found = 1, config missing/corrupt = 2, plan missing for apply = 3). Loud-on-issues is v0.2 `check`'s job. See §8.

No new open questions surfaced by the v1 review. Next review pass may identify others.
